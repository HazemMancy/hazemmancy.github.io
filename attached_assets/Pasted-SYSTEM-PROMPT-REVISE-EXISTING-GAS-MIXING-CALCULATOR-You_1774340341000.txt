MANDATORY TECHNICAL CORRECTIONS
--------------------------------------------------

1) ENFORCE NORMALIZATION POLICY IN THE SOLVER — CRITICAL
Current issue:
Normalization policy (strict vs normalize) is handled only at page level, while the solver always normalizes.

Required correction:
Make normalization mode an explicit solver input / calculation policy and enforce it inside the actual calculation logic.

Required behavior:
- strict mode:
  - reject compositions whose mole fractions do not sum to 1.0 within tolerance
- normalize mode:
  - normalize the composition intentionally and report that normalization occurred

Do NOT let the solver silently normalize when the selected mode is strict.

2) FIX MULTI-STREAM COMPOSITION HANDLING — CRITICAL
Current issue:
In multi-stream mixing, each stream composition can be off-sum and still be used with only a warning.

Required correction:
For each stream, implement explicit composition policy handling.

Preferred implementation:
- apply per-stream normalization policy consistent with selected mode:
  - strict → reject invalid stream composition
  - normalize → normalize each stream before mixing and report it

At minimum:
- do not continue with materially invalid stream compositions using warning-only logic

This is mandatory for result integrity.

3) HANDLE STREAM FLOW UNIT HONESTLY — CRITICAL
Current issue:
The multi-stream data model includes flowUnit, but the solver does not actually use it.

Required correction:
Implement one of the following, preferred in this order:

OPTION A — PREFERRED
Implement real per-stream flow-unit conversion before mixing.
Supported units must be clearly defined and converted into a single consistent molar-flow basis before blending.

OPTION B
If unit conversion is not implemented now:
- remove flowUnit from the active user-facing model and UI
- clearly state that all stream flows must be entered on one consistent molar-flow basis

Do NOT keep a dead engineering input that implies unsupported unit flexibility.

4) STRENGTHEN APPROXIMATE PROPERTY DISCLOSURE — CRITICAL
Current issue:
The calculator estimates gamma, Cp/Cv, viscosity, Z, JT, and speed of sound using simplified methods, but the UI can be over-read as more rigorous than it is.

Required correction:
Methodology / assumptions / warnings must clearly state that operating-condition properties are:
- screening / approximate only
- not a substitute for full EOS/property-package calculations
